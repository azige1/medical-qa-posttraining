# 项目主任务规格

## 1. 最终定位

第一版项目主任务固定为：

**中文 Text-to-SQL / 数据分析问答后训练**

它的目标不是做一个通用聊天模型，也不是做一个 SQL Agent，而是训练一个在给定数据库 schema 的前提下：

- 更稳定地产生可执行 SQL
- 更准确地完成 schema grounding
- 更少出现错表、错列、错过滤条件
- 更少输出危险或越权 SQL

的大模型后训练系统。

## 2. 对外表述

项目对外统一表述为：

- 中文 Text-to-SQL 后训练项目
- 基于 MedicalGPT 脚手架的大模型 SQL 生成项目

不使用以下表述：

- 数据分析 Agent
- 数据库自动运维系统
- 多工具 SQL 助手

因为第一版不做 agent 编排，也不做工具调用闭环。

## 3. 任务边界

### 3.1 第一版纳入范围

- 单轮中文问题到单条 SQL 的映射
- 输入包含自然语言问题和完整 schema_text
- 输出限定为 SQLite 方言的单条 `SELECT`
- 重点覆盖过滤、聚合、排序、分组、简单 join、top-k

### 3.2 第一版明确不做

- 多轮澄清式问答
- DDL / DML / 多语句 SQL
- 跨数据库方言混训
- 解释型回答或 SQL+自然语言混合输出
- 工具调用式 agent 闭环

## 4. 输入与输出

### 4.1 输入

单条样本由两部分组成：

- `schema_text`
- `question_zh`

典型例子：

- 给定销售库 schema，问“每个城市客户的订单总金额是多少？”
- 给定教务库 schema，问“2024 春季学期平均分最高的课程是哪门？”

### 4.2 输出

输出固定为单条 SQL，不附加解释，不输出 Markdown，不输出代码块。

详细格式见 [`sql_output_spec.md`](sql_output_spec.md)。

## 5. 优化优先级

第一版模型的优化顺序固定为：

1. 执行结果正确
2. 表列引用正确
3. SQL 可执行且只读
4. 语句尽量简洁稳定
5. 尽量减少格式噪声和解释性文本

以下因素不是第一优先级：

- SQL 写法是否最“优雅”
- 输出是否带自然语言说明
- 是否支持复杂 agent 交互

## 6. 为什么这个任务最合适

这个主任务同时满足四个条件：

1. **数据可得**
   - `CSpider` 和 `Spider` 提供了稳定的 Text-to-SQL 训练来源。
2. **训练闭环清晰**
   - `SFT + DPO` 能直接优化“更正确的 SQL 比更错误的 SQL 更好”。
3. **评测可执行**
   - 可以用官方 `CSpider dev` 和自建 SQLite 小库同时验证。
4. **面试可讲**
   - 很容易把数据构建、训练目标和执行评测串成一个完整项目故事。

## 7. 与训练链路的关系

- `SFT`：先让模型学会“看 schema 写 SQL”的基本行为分布
- `DPO`：用 preference pair 进一步压错表、错列、错条件
- `GRPO`：通过可执行 reward 直接优化 valid SQL / execution accuracy
- `RM + RLOO/PPO mini`：作为扩展实验，用于证明理解完整 RLHF 链路
